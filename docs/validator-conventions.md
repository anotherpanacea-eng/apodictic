# Validator-Conventions ‚Äî a meta-linter that validates the validator fleet

**Status:** **Built (Increment 1), 2026-06-20.** Shipped: `scripts/meta_lint.py` + `validate.sh validator-conventions` (M1‚ÄìM5), wired into `--self-test-all` and `--check-all`. A validator that validates the validators.
<!-- built-when: scripts/meta_lint.py -->

APODICTIC's validator fleet (count **derived** from `AGG_VALIDATORS`, never hand-typed ‚Äî so this prose names no integer) shares a set of structural conventions ‚Äî every self-testable validator is wired into the dispatcher with a `--self-test`; file resolvers classify inputs on **parsed blocks**, never a raw `apodictic:<type>` marker substring; the advertised count is derived; every schema is actually consumed; override markers are detected with the hardened boundary-matched helper, never a bare substring. Until now these were enforced only by manual review, and the gaps caused real latent bugs: a fleet-wide adversarial sweep (2026-06-20) found a **resolver-substring** class in 13 validators (a file that merely *names* a marker in prose misroutes and silently false-passes) and an **override-substring** class in every override-honoring gate (a suffixed slug or a code-span decoy is wrongly honored), plus crash-on-malformed and lexical-regex over-fire classes. The substring classes are *structural* and statically checkable ‚Äî so this validator mechanizes them (M2 + M5), and they cannot silently re-enter.

## The checks

`validate.sh validator-conventions` reads `validate.sh` and the sibling `*.py` from its own directory (and the schemas via the shared resolver). No artifact input.

| ID | Rule |
|---|---|
| **M1 ‚Äî dispatch + self-test** | Every name in `AGG_VALIDATORS` has a `<name>)` dispatcher case whose body handles `--self-test` (so `--self-test-all` can run it). A missing case or a case that ignores `--self-test` is a violation. |
| **M2 ‚Äî resolver hygiene** | No validator script classifies inputs by a raw `apodictic:<type>` marker scan ‚Äî a literal `apodictic:<type>` (DIGITS allowed; an f-string `f"apodictic:{bt}"` counts) reached by a membership op (`in` / `not in`) OR a str/re scan op (`.find` / `.count` / `.index` / `.startswith` / `re.search` / `re.findall` / ‚Ä¶). The anti-pattern misroutes a file naming the marker in prose. Resolvers must classify on parsed blocks (the `_has_block` / `art.parse_blocks` idiom). The safe `_has_block` degraded fallback uses a `"apodictic:%s" % btype` **format** string, which the literal pattern deliberately does not match. Scanned with comments/docstrings stripped, so a marker named in a comment, a docstring, or an error-message string is not a false positive. (Infra scripts with no resolver are exempt: `apodictic_artifacts.py`, `meta_lint.py`, `sync_setec.py`, `config_checks.py`.) |
| **M3 ‚Äî derived count** | `validate.sh` derives the count from `AGG_VALIDATORS` (`AGG_COUNT=$(set -- $AGG_VALIDATORS; echo $#)`) ‚Äî never a hand-typed integer that can drift (every Horizon spec written before 2026-06-20 wrongly told builders to "bump the 35‚Üí36 count string"). |
| **M4 ‚Äî no orphan schema** | Every `*.schema.json` in the resolved schema dir has its **filename stem** (what `art.known_schema_ids()` returns ‚Äî not a parsed `$id`) referenced as a **whole id** (boundary-matched, so a referenced `‚Ä¶v10` does not mask an orphan `‚Ä¶v1`) by at least one validator script ‚Äî a schema nothing consumes is dead or mis-wired. If the resolver can't be imported or resolves zero schemas, M4 emits a **WARN** (the check is skipped, not asserted clean) rather than silently passing. |
| **M5 ‚Äî override hygiene** | No validator detects an override marker by a bare `<!-- override: <slug>` substring scan ‚Äî Python `"<!-- override: %s" % slug in body` / `"<!-- override: x" in body` / `re.findall(r"<!-- override: ‚Ä¶")`, or bash `grep -F "<!-- override:`. That bare-prefix form honors a SUFFIXED slug (`<slug>-but-not-really`) and a code-span decoy (a marker quoted inside a backtick span / fenced block). Override detection must go through the shared **`override_marker.has_override`** (Python) / **`_has_override`** (bash) helper, which strips code spans (fenced + inline) and boundary-matches the slug. The override-marker sibling of M2's resolver-substring class. (`override_marker.py` ‚Äî the helper ‚Äî and `meta_lint.py` are exempt.) |

`meta_lint.py --self-test` exercises each check function against synthetic inputs (always green); the `--check-all` live run validates the actual fleet (gating it at release). It is itself in `AGG_VALIDATORS`, so **M1 checks the meta-linter against its own rule.**

## Not (yet) mechanized

The **override-substring** class is now mechanized by **M5** and fixed fleet-wide (the bare-marker gate-bypass in every override-honoring Python and bash gate, not just `softness-check` / `timeline_checks`): all sites call the shared hardened `override_marker.has_override` / `_has_override` helper, and M5 gates the bare form going forward (see the changelog for the full site list and the bash‚ÜîPython fenced-block-divergence fix). The sweep's REMAINING classes stay out of scope for the meta-linter (M1‚ÄìM5) because they need per-validator semantics, not a structural grep ‚Äî though they are nonetheless **fixed per-validator in this same PR**, just not statically gated. **Crash-on-malformed** (a non-dict block payload reaching `obj.get()`, plus the non-string-`id` variant in `softness-check`) is fixed per-validator with `isinstance(...)` guards (the companion crash work in this PR). **Lexical-regex over/under-fire** (heuristic-tuning the specs already concede is fuzzy) stays a tracked follow-up. A future increment could add a crash-resistance check (every `parse_*` guards non-dict before `.get()`), but that needs AST-level analysis rather than the line-level greps M1‚ÄìM5 use.
